# 第5讲：MOESI 协议的硬件实现

## 本讲概要

前面几讲我们从协议层⾯理解了 MOESI 的五种状态及其转换。本讲我们下沉到硬件层⾯：状态位存储在哪⾥？Tag 与 Data 为何分开？监听过滤器如何⼯作？⽬录协议如何替代⼴播？这些都是⾯试中的⾼频考点，同时也是 CPU 微架构师⽇常打交道的内容。

---

## 1. 状态位存哪⾥：Tag SRAM 结构

MOESI 的状态位并⾮存储于某个独⽴的寄存器堆中，⽽是**紧贴在 Tag 旁边，共同组成 Tag SRAM 的⼀个条⽬（entry）**。每个 cache line 在 Tag SRAM 中映射到⼀个包含了地址、状态、合法位、脏位、替换信息等元数据的结构体。

```
+-- Set 0 --------------------------------------------------------------+
| Way 0 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 1 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 2 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 3 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 4 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 5 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 6 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
| Way 7 | Tag[33:0] | S[2:0] | V | D | LRU[2:0] |                     |
+----------------------------------------------------------------------+
| 字段        | 位宽     | 说明                                       |
|-------------|----------|-------------------------------------------|
| Tag[33:0]   | 34 bits  | 物理地址的 tag 部分                         |
| S[2:0]      | 3 bits   | MOESI 状态编码（M/O/E/S/I）                 |
| V           | 1 bit    | Valid bit — 该 cache line 是否包含有效数据    |
| D           | 1 bit    | Dirty bit — 数据是否被修改过（与 S 位部分冗余）|
| LRU[2:0]    | 3 bits   | 伪 LRU 替换策略的状态位（8-way 需要）         |
+----------------------------------------------------------------------+
```

**关键设计点：**
- 整个 Tag SRAM 是按 **set 为单位组织**的。硬件索引到某个 set 时，该 set 的全部 8 路 Tag + State 同时被读出。
- 状态位和脏位存储在 Tag 端，⽽⾮ Data 端，因为 **状态查询（tag lookup + state check）不需要读取 Data SRAM** — 这对监听（snoop）性能⾄关重要。

---

## 2. 状态编码：3 位编码 5 种状态

MOESI 有 5 种有效状态：**M, O, E, S, I**。为什么需要 3 位？因为 $2^2 = 4 < 5$，必须使⽤ 3 位来区分全部 5 种状态。剩下的 3 种编码（8 - 5 = 3）可以视为非法状态，或⽤于调试/故障检测。

```
+-------+--------+------------------------------+
| S[2]  | S[1:0] | 状态                         |
| (Dirty)|(编码) |                              |
+--------+-------+------------------------------+
|  0    |  00   | I (Invalid) — 000            |
|  0    |  01   | S (Shared)  — 001            |
|  0    |  10   | E (Exclusive)— 010            |
|  0    |  11   | (保留/⾮法) — 011             |
|  1    |  00   | M (Modified)— 100            |
|  1    |  01   | (保留/⾮法) — 101             |
|  1    |  10   | (保留/⾮法) — 110             |
|  1    |  11   | O (Owned)   — 111            |
+--------+-------+------------------------------+
```

**⾯试要点 — 为什不⽤ 2 位？**
- 如果只⽤ 2 位，7 种状态需要 3 位（$2^2 = 4$ 不够）。5 种状态同样需要 3 位（$2^2 = 4 < 5$）。
- 如果你的 CPU 仅实现 MESI（4 种状态），2 位刚够。但 MOESI 多了 O 状态，必须升级到 3 位。
- 额外的好处：S[2] 恰好对应 dirty 语义（M 和 O 都置位），S[1] 区分 exclusive 属性（E 和 M），这在组合逻辑中可以简化状态比较。

## 3. Tag SRAM 与 Data SRAM：为何分离

```
+---------------------------+        +---------------------------+
|       TAG SRAM            |        |       DATA SRAM           |
|                           |        |                           |
| Set 0: Way0 Way1 ... Way7 |        | Set 0: Way0 Way1 ... Way7 |
| Set 1: Way0 Way1 ... Way7 |        | Set 1: Way0 Way1 ... Way7 |
| Set 2: Way0 Way1 ... Way7 |        | Set 2: Way0 Way1 ... Way7 |
|   ...                     |        |   ...                     |
| Set N: Way0 Way1 ... Way7 |        | Set N: Way0 Way1 ... Way7 |
+---------------------------+        +---------------------------+
       |     |                              |     |
       |     +-- 标签⽐较                   |     +-- 数据选择
       |            |                       |            |
       v            v                       v            v
   [Comparator x8]               [Data Mux 8:1]
```

**物理分离的原因有三：**

1. **独立访问需求**：监听（snoop）请求只需要读取 Tag SRAM 来做地址匹配和状态查询，完全不需要 Data SRAM 参与。如果 Tag 和 Data 合并在⼀起，每次监听请求都会浪费读取⼀整条 cache line 的功耗（32/64 字节 vs 几十位 tag）。

2. **不同的端⼝需求**：Tag SRAM 需要多个读端⼝（CPU 侧 + 监听侧），⽽ Data SRAM 端⼝数相对较少。分离后可以独⽴优化每部分的端⼝设计。增加一个 SRAM 端⼝的⾯积代价是非常高的。

3. **不同的宽度与时序**：Tag SRAM 宽度窄（44 bits 左右），Data SRAM 宽度宽（512 bits / 64 bytes）。窄 SRAM 读出速度更快，可以在数据读出之前完成 tag ⽐较。这形成了⾃然的**流水线时序** — Tag 先较，命中后才选中对应 way 的 Data 输出。

**面试要点：** Tag SRAM 决定了缓存的命中延迟，Data SRAM 决定了带宽。

## 4. 组相联查找的硬件实现（全流程详解）

以下是一个 8 路组相联 L1 缓存的 **单周期** 查找全过程。所有操作在⼀个时钟周期内完成。

```
输入：物理地址 addr[47:0]
+--+--------------------------+------------------+
|  |        Tag (34b)         |   Index (10b)   | Offset (6b) |
+--+--------------------------+------------------+


Step 1: Index 译码 — 选中 Set
                                 Index[9:0]
                                     |
                            +--------v--------+
                            |  3:8 Decoder    |
                            | (1-hot 选择)    |
                            +--------+--------+
                                     |
                             选中 Set 的 wordline
                                     |
        +----------------------------+----------------------------+
        |                            |                            |
        v                            v                            v
Step 2: 并行读出 Tag + State    Step 2 (同时):                 Step 2 (同时):
+----------------------+     +----------------------+     +----------------------+
| Way 0: Tag[33:0]    |     | Way 1: Tag[33:0]    |     | Way 7: Tag[33:0]    |
|        S[2:0]       |     |        S[2:0]       |     |        S[2:0]       |
|        V             |     |        V             |     |        V             |
+---------+------------+     +---------+------------+     +---------+------------+
          |                            |                            |
          v                            v                            v
Step 3: 8 路并行⽐较 (XOR + AND tree)
+------------------+    +------------------+         +------------------+
| addr[47:14]      |    | addr[47:14]      |         | addr[47:14]      |
|      ==          |    |      ==          |  ....   |      ==          |
| Way0_Tag[33:0]   |    | Way1_Tag[33:0]   |         | Way7_Tag[33:0]   |
+--------+---------+    +--------+---------+         +--------+---------+
         |                       |                            |
   match[0]                 match[1]                    match[7]
         |                       |                            |
         +-----------+-----------+--------.......------------+
                     |
                     v
Step 4: 命中逻辑 (AND with Valid)
   hit[0] = match[0] & Way0_V
   hit[1] = match[1] & Way1_V
   ...
   hit[7] = match[7] & Way7_V
                     |
                     v
              +------+------+
              | Priority /  |
              | One-hot     |  (只有1路命中; 多路命中=硬件错误)
              +------+------+
                     |
                way_select[2:0]  (3位编码 0-7)
                     |
                     v
Step 5: 数据选择
   Data SRAM (同⼀ Index, 8路同时读出)
   +------------+
   | Way0_Data  |---+
   | Way1_Data  |---+
   | Way2_Data  |---+---[8:1 MUX]----> 最终数据 Data[511:0]
   | Way3_Data  |---+      |
   | Way4_Data  |---+  way_select
   | Way5_Data  |---+
   | Way6_Data  |---+
   | Way7_Data  |---+
   +------------+

Step 6: 状态/命中判断
   cache_hit = (hit[0] | hit[1] | ... | hit[7])
   if !cache_hit: 触发 cache miss → FSM 启动填从下级存储填充
   当前状态 = 选中 way 的状态位 (S[2:0])
```

**为什么能做到单周期？**
- 所有 8 路 tag 读操作和 8 路 data 读操作是**全并行的**。
- ⽐较器使用简单的 XOR + AND 链，路径深度仅 log₂(34) ≈ 6 级门延迟。
- Data MUX 是纯组合逻辑，选择信号在 tag 比较完成时立即生效。
- 对于高速 L1（如 4-5 GHz），这个路径可能仍然太慢，此时会采用 **way predictor** 来提前猜测要读哪一路。

## 5. Cache Controller FSM（缓存控制器有限状态机）

缓 存控制器是一块**硬连线逻辑（hardwired logic）**，不是运行在微码引擎或软件上。它由几百个逻辑门组成，直接通过组合逻辑实现状态转移。

```
                      +-----------------------------+
                      |                             |
    cpu_req ─────────>|    Next State Logic         |
    snoop_req ───────>|    (LUT + 组合逻辑)         |
    current_state ───>|                             |
    fill_done ───────>|   ~200-500 gates            |
    evict_done ──────>|                             |
                      +-------------+---------------+
                                    |
                                    | next_state[2:0]
                                    v
                      +-------------+---------------+
                      |                             |
                      |   Current State Register    |
                      |   (3-bit Flip-Flop)         |
                      |                             |
                      +-------------+---------------+
                            |               |
                   current_state       输出到
                            |          Tag SRAM
                            v
                      +-------------+---------------+
                      |                             |
                      |   Output Logic (Mealy)       |
                      |                             |
                      |  - tag_write_en              |
                      |  - data_write_en             |
                      |  - state_write_en            |
                      |  - snoop_response[1:0]       |
                      |  - fill_request              |
                      |  - evict_request             |
                      |  - cache_hit                 |
                      |                             |
                      +-----------------------------+
```

**FSM 状态机的两种模型：**

- **Moore 型**：输出仅取决于当前状态。输出逻辑更简单，但响应延迟多一个周期。
- **Mealy 型**：输出取决于当前状态 + 输入。响应更快（组合逻辑路径），但时序可能更紧张。

现代实现通常使⽤ **Mealy 型**处理 snoop_response 和 fill_request（需要快速响应），⽤ **Moore 型**处理状态更新。

**FSM 并非软件！**
```
// 这不是 C 代码在 CPU 上跑 — 这是直接综合为门的 Verilog:
always @(posedge clk or negedge rst_n) begin
    if (!rst_n)
        current_state <= I;
    else
        current_state <= next_state;
end

always @(*) begin
    case (current_state)
        I:  if (cpu_read)   next_state = (other_core_has_copy) ? S : E;
        S:  if (cpu_write)  next_state = M;
        E:  if (cpu_write)  next_state = M;
        M:  if (snoop_read) next_state = O;
        O:  if (cpu_write)  next_state = M;
    endcase
end
// 综合后 = ~150 个与非门 + 3 个触发器
```

## 6. Snoop Filter（监听过滤器）

### 6.1 问题：为何需要监听过滤器？

在⼀个 64 核处理器中，如果每次总线上发⽣事务，所有 64 个核的 Tag SRAM 都要被查找，那么：
- 64 个核 × 每个核的 Tag 读功耗 = 巨⼤的功耗开销
- Tag SRAM 端⼝被监听请求占用，与 CPU 的读写请求形成竞争

**监听过滤器的核⼼思想：**维护一个紧凑结构，在 Tag SRAM 访问以前就过滤掉绝大部分无关的监听请求。

### 6.2 布隆过滤器（Bloom Filter）

布隆过滤器是最经典的硬件监听过滤器实现。

```
原：每个监听请求 → 所有 64 核 → 全查 Tag SRAM

有：每个监听请求 → Bloom Filter 查询 → "Definitely NOT present" (90% 的请求在此过滤)
                                       → "Maybe present" → 查 Tag SRAM (仅 10%)
```

**布隆过滤器工作原理：**

```
Insert: Core X loads cache line at address A
    addr_A ──→ [Hash1] ──→ 位向量索引 3   ──→ set bit[3]
         ──→ [Hash2] ──→ 位向量索引 17  ──→ set bit[17]
         ──→ [Hash3] ──→ 位向量索引 41  ──→ set bit[41]

Lookup: 收到对地址 A 的监听请求
    addr_A ──→ [Hash1] ──→ bit[3]  = 1  ✓
         ──→ [Hash2] ──→ bit[17] = 1  ✓
         ──→ [Hash3] ──→ bit[41] = 1  ✓
    → 全部为 1 → "Maybe present" → 需要查 Tag SRAM

Lookup: 收到对地址 B 的监听请求（从未被缓存）
    addr_B ──→ [Hash1] ──→ bit[5]  = 0  ✗
         ──→ [Hash2] ──→ bit[22] = 0  ✗
         ──→ [Hash3] ──→ bit[50] = 1  ✓
    → 存在 0 → "Definitely NOT present" → 直接忽略，不查 Tag SRAM
```

**布隆过滤器的关键特性：**
- **假阳性（False Positive）：** 可能报告 "Maybe present"，即使该地址不在缓存中
- **假阴性（False Negative）：不可能** — 绝不漏报。这是布隆过滤器的理论保证。
- **无需删除：** 当 cache line 被替换出去时，不清理布隆过滤器的对应位。过滤器定期全部清零重建。

**硬件开销：** 一个 1 Kbit 的布隆过滤器（约 128 字节）可以覆盖 256 条 cache line 的跟踪，假阳性率 ≈ 1~2%。

### 6.3 共享者表（Sharer Table）

另一种替代方案是维护共享者表 — 精确跟踪但代价略高：

```
+------------------+----------------------------------+
| Cache Line Addr  |  Sharer Bitmap (1 bit / core)   |
+------------------+----------------------------------+
| 0xABCD0000       | 0000_0000_0000_0001 (Core 0)     |
| 0x12340000       | 0000_0100_0000_0011 (Core 0,1,10)|
+------------------+----------------------------------+
```

共享者表查询精确，没有假阳性，但 SRAM 开销远大于布隆过滤器。

## 7. 目录协议（Directory Protocol）

### 7.1 广播式监听的局限性

在广播式（snooping）协议中，每个 cache miss 都需要通过总线广播给所有核心。当核心数上升到 32、64 或更多时，总线的串行化开销和功耗变得不可接受。

### 7.2 目录协议的基本原理

**核心思想：**用一个中央目录跟踪每条 cache line 被哪些核心持有，然后将监听消息点对点发送，而非广播。

```
                   +---------------------------+
                   |     Directory (在 L3 或    |
                   |     独立目录缓存中)        |
                   |                           |
                   |  Addr     | Sharer Bitmap  |
                   |-----------+--------------- |
                   |  Line A   | 0000_0011      | ← Core 0,1 持有 S 态副本
                   |  Line B   | 0000_0001      | ← Core 0 持有 E/M 态
                   |  Line C   | 0000_0000      | ← 无核心持有
                   +---------------------------+
                        |          |         |
               点对点消息↓  点对点消息↓   无消息(该地址无副本)
                   +----+    +----+     +----+
                   |Core|    |Core|     |Core|
                   | 0  |    | 1  |     | 2  |
                   +----+    +----+     +----+
```

**对比：广播 vs 目录**

```
广播式 (Snooping):
    Core 0 miss → 广播到 Core1, Core2, ..., Core63 (64 个 Tag lookup)
    总线利用率 = 高, 扩展性 = 差

目录协议 (Directory):
    Core 0 miss → 查目录 → Sharer Bitmap = 0000_0011
                → 仅向 Core 0 和 Core 1 发送 snoop (2 个 Tag lookup)
    总线利用率 = 低, 扩展性 = 好
```

**目录条目的数据结构：**

```
+------------------+--------+--------+------------------------+
| Tag (地址标签)    | State  | Owner  | Sharer Bitmap (每核1bit)|
+------------------+--------+--------+------------------------+
| 34 bits          | 3 bits | 6 bits | N bits (N = 核心数)    |
+------------------+--------+--------+------------------------+

State:
  - UNCACHED (U): 无核心持有
  - SHARED (S):   多个核心持有 S 态副本
  - EXCLUSIVE (E):单个核心持有 E 或 M 态
```

**目录的三种典型位置：**

1. **与 L3 合并在⼀起：** 每个 L3 cache line 附带一个 sharer bitmap。这是最经济的实现。
2. **独立目录缓存：** 专门的 SRAM 结构，只存目录信息。L3 不需要时更有优势。
3. **分布式目录：** 每个内存控制器维护一部分地址空间的目录，适用于 NUMA 架构。

## 8. Inclusive vs Non-Inclusive 缓存

### 8.1 Inclusive Cache（包含缓存）

在包含缓存设计中，L3 包含所有 L1/L2 中存在的 cache line。这意味着 L3 天然具备 "Core Valid Bit" 信息。

```
+------- L3 Cache Line ------+
| Tag | Data | CoreValid Bits |
|     |      | C0 C1 C2 C3 .. |
+-----------------------------+

CoreValid Bits: 每个 bit 对应一个核心。
  如果 CoreValid[i] = 1 → Core i 的私有缓存中可能存在该 cache line
  如果 CoreValid[i] = 0 → Core i 的私有缓存中肯定没有该 cache line
```

**L3 作为天然的监听过滤器：**

```
收到对地址 A 的监听请求 →
  查 L3 → 找到对应 line
    → CoreValid Bits = 0000_0101  (即 Core 0 和 Core 2 可能有)
    → 仅向 Core 0 和 Core 2 发送监听消息
    → Core 1,3,4,5,6,7 完全不受打扰
```

**优点：** L3 天然就是监听过滤器，不需额外硬件。
**缺点：** L3 必须大于所有私有缓存之和，否则不断发生 L3 逐出并连带冲刷私有缓存。

### 8.2 Non-Inclusive Cache（非包含缓存）

非包含设计中，L3 不保证包含私有缓存的内容。一条 cache line 可能存在于 L1，但不在 L3 中。

**优缺点：**
- 优点：L3 容量利用率更高，不强制存储已经存在于私有的行。
- 缺点：失去了天然监听过滤能力，需要额外的监听过滤器硬件。

### 8.3 Exclusive Cache（独占缓存）

独占缓存在 L1/L2 和 L3 之间不存储重复数据。当一条 line 从 L3 被调入 L1 时，L3 中的副本被无效化。在 MOESI 语境中较少使用，更多见于 AMD 的某些微架构设计。

## 9. Cache-to-Cache 传输的数据路径

当 Core 0 持有 M 态数据，Core 1 对该地址发起了读取请求，数据直接从 Core 0 的 L1 传给 Core 1。下面是物理数据路径的逐级描述。

### 9.1 数据路径（逐级）

```
Core 0 (数据源 — Owner)                            Core 1 (数据⽬标 — Requestor)
+---------------------------+                      +---------------------------+
|                           |                      |                           |
| L1 Data SRAM              |                      | L1 Data SRAM              |
|   Way 3 (命中)             |                      |   Way 5 (待填充)           |
|   64 bytes                |                      |   64 bytes                |
|       |                   |                      |       ^                   |
|       v                   |                      |       |                   |
| [Sense Amps]              |                      |  [Write Drivers]          |
|       |                   |                      |       |                   |
|       v                   |                      |       |                   |
| [Output Latch / Buffer]   |                      | [Input Latch / Buffer]    |
|   512 bits                |                      |   512 bits                |
|       |                   |                      |       ^                   |
+-------+-------------------+                      +-------+-------------------+
        |                                                |
        v                                                |
+-------+-------------------+                            |
| [Parity / ECC Encode]     |                            |
+-------+-------------------+                            |
        |                                                |
        v   +-------+ 64B/trans                         |
+-------+---+-------+------ On-Chip Data Bus ------+----+-------------------+
        16 bytes per beat (128-bit bus), 4 beats = 64 bytes
        每拍 16 字节, 共 4 拍传完一条 cache line
```

### 9.2 延迟对比

```
操作                         | 典型延迟 (周期 @ 3GHz) | 典型延迟 (ns)  |
-----------------------------|-----------------------|----------------|
L1 Cache Hit                 | 4 周期                | ~1.3 ns        |
L2 Cache Hit                 | 12 周期               | ~4 ns          |
Cache-to-Cache Transfer      | 10-15 周期            | ~3-5 ns        |
L3 Cache Hit                 | 40 周期               | ~13 ns         |
Local DRAM Read              | ~200 周期             | ~67 ns         |
Remote Socket DRAM (NUMA)    | ~400 周期             | ~133 ns        |
```

**关键结论：** Cache-to-cache 传输比 DRAM 读取快约 **15~20 倍**。这正是 MOESI 协议中 O 状态和 M→O 状态转换的设计动⼒ — 通过缓存间传输来避免访问主存。

### 9.3 为什么 Cache-to-Cache Transfer 快？

1. **物理距离短：** 数据在芯片内两个核心之间传输，走的是片上网络或共享总线，无需离开芯片封装。
2. **不用经过内存控制器：** 省去了将请求编码为 DRAM 命令（ACTIVATE → READ → PRECHARGE）的开销。
3. **总线宽度大：** 片上数据总线通常为 128-256 bit，而 DRAM 通道为 64 bit（DDR4）或 32 bit（每个 channel）。

## 10. 测验

### Q1：为什么 MOESI 需要 3 位而不是 2 位来编码状态？

**参考答案：**
2 位只能编码 4 种状态（$2^2 = 4$，即 00, 01, 10, 11）。MOESI 有五种有效状态（M, O, E, S, I），$4 < 5$，所以最少需要 3 位编码。与之对比，MESI 协议仅使用 4 种状态，理论上 2 位就足够。

---

### Q2：监听过滤器（Snoop Filter）如何减少不必要的 Tag 查询？

**参考答案：**
监听过滤器维护一个紧凑的数据结构（如布隆过滤器或共享者表），在访问 Tag SRAM 之前快速判断某个地址是否可能存在于本地缓存中。如果过滤器返回 "Definitely NOT present"，则直接忽略该监听请求，完全不需要访问 Tag SRAM。在 64 核系统中，这可以将有效 Tag 查询量降低 80-95%，大幅降低功耗并减轻 Tag SRAM 的端口争用。

---

### Q3：在目录协议中，某核心发起对地址 0xABCD 的读取请求，目录发现该行被 Core 3 和 Core 7 以 S 态持有。此时会发生什么（而不是向全部 64 核广播）？

**参考答案：**
目录协议不会向全部 64 个核心广播。目录控制器查看该 cache line 对应的 Sharer Bitmap（位图值为 `0000_0000_0000_..._1000_1000`），仅向 Core 3 和 Core 7 发送点对点的监听消息（无效化/降级/数据请求），其余 61 个核心完全不受影响。

---

### Q4：为什么 Tag SRAM 和 Data SRAM 要作为独立的物理结构？（列举至少两个原因）

**参考答案：**

1. **监听只需访问 Tag：** 监听请求只需要查看地址标签和状态信息，无需读取 64 字节的完整数据。如果 Tag 和 Data 合并存储，每次监听都会浪费读取 512 bits 数据的功耗。
2. **可以独立优化端口数量：** Tag SRAM 需要更多的读端口（CPU 访问 + 监听请求），而 Data SRAM 端口数较少。分离后可以独立调整每部分的端口数，降低总体面积。
3. **时序优化：** Tag SRAM 很窄（约 40-50 bits），输出快于 Data SRAM。先完成 tag 比较再选择 data 输出，形成自然的流水线关键路径。

---

### Q5：Cache-to-Cache 传输和一次 DRAM 读取的延迟对比大约是多少？为什么差距这么大？

**参考答案：**
Cache-to-cache 传输约 **10-15 个周期**（~3-5 ns @ 3GHz），而 DRAM 读取约 **200 个周期**（~67 ns）。差距约 15-20 倍。

原因：
1. 数据在芯片内部传输，不经外部引脚，延迟极低。
2. 不需要经过 DRAM 的行激活（ACTIVATE）、列读取（READ）、预充电（PRECHARGE）等复杂时序。
3. 片上数据总线更宽（128-256 bits vs DDR4 的 64 bits），一个 cache line 的传输拍数更少。
