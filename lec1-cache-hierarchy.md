## 1. 为什么需要缓存 —— 存储墙问题

### 1.1 CPU 与 DRAM 的速度剪刀差

现代 CPU 的主频通常在 3-5 GHz，即一个时钟周期约 **0.2-0.3ns**。而访问一次主存（DRAM）的延迟大约是 **100ns**。这意味着：

```
CPU 执行一条指令:     ~0.3ns
DRAM 一次访问:       ~100ns  (等待 300+ 个时钟周期)
```

这个差距被称为 **存储墙（Memory Wall）**。

### 1.2 各级缓存的典型延迟

以 Intel/AMD 现代 x86 处理器为参考（约 4GHz 主频）：

| 层级 | 典型大小 | 延迟（近似） | 时钟周期 |
|------|---------|-------------|---------|
| L1 Cache | 32KB (I) + 32KB (D) per core | ~1 ns | ~4 cycles |
| L2 Cache | 256KB - 512KB per core | ~5 ns | ~12 cycles |
| L3 Cache (LLC) | 8MB - 32MB shared | ~20 ns | ~40 cycles |
| DRAM | 8GB - 64GB | ~100 ns | 200+ cycles |

关键洞察：**延迟每下降一级，差距约为 3-5 倍**。如果数据在 L1 中命中，CPU 几乎不需要等待；如果必须访问 DRAM，CPU 将空转 200+ 个周期。

### 1.3 局部性原理

缓存之所以有效，是因为程序行为遵循两条规律：

- **时间局部性（Temporal Locality）**：刚被访问过的数据大概率很快会再次被访问（例如循环中的变量）。
- **空间局部性（Spatial Locality）**：地址附近的数据大概率也会被访问（例如数组遍历、连续的结构体字段）。

> **面试要点**：能用自己的话解释为什么局部性让缓存有效，并能举出具体的代码例子（如 `for` 循环遍历数组 vs 随机访问链表）。

## 2. 缓存层次结构

### 2.1 典型多核系统的缓存拓扑

```
                          +-----------------------+
                          |        DRAM            |
                          |   (8GB - 64GB)         |
                          +-----------+-----------+
                                      |
                          +-----------+-----------+
                          |   L3 Cache (LLC)       |
                          |   8MB - 32MB, shared   |
                          +---+----+----+----+---+
                              |    |    |    |
                  +-----------+    |    |    +-----------+
                  |                |    |                |
          +-------v------+  +-----v----v-----+  +-------v------+
          |    L2 Cache   |  |    L2 Cache    |  |    L2 Cache   |
          |   256KB pvt   |  |   256KB pvt    |  |   256KB pvt   |
          +-------+-------+  +-------+--------+  +-------+-------+
                  |                  |                   |
          +-------v-------+  +-------v-------+   +-------v-------+
          | L1I  |  L1D    |  | L1I  |  L1D   |   | L1I  |  L1D   |
          | 32KB | 32KB    |  | 32KB | 32KB   |   | 32KB | 32KB   |
          +-------+--------+  +-------+-------+   +-------+-------+
                  |                  |                   |
          +-------v-------+  +-------v-------+   +-------v-------+
          |    Core 0     |  |    Core 1     |   |    Core 2     |
          +---------------+  +---------------+   +---------------+
```

### 2.2 私有 vs 共享

| 属性 | L1 Cache | L2 Cache | L3 Cache (LLC) |
|------|---------|---------|-----------------|
| 所属关系 | 每个核私有 | 每个核私有（通常） | 所有核共享 |
| 延迟 | 最低 | 中等 | 较高 |
| 大小 | 最小 | 中等 | 最大 |
| 一致性影响 | 直接导致一致性问题 | 同样需要一致性 | 作为一致性后盾 |

- **L1 和 L2 是私有的**：这意味着 Core 0 的 L1 中有一份数据，Core 1 的 L1 中可能有一份同一地址的"副本"。如果它们不一致，就产生了 **缓存一致性问题**。
- **L3 (LLC) 是共享的**：Last Level Cache，所有核心都能访问。它通常也是 CPU 芯片上最后一级缓存，再往外就是内存控制器和 DRAM。

### 2.3 Inclusive vs Non-Inclusive vs Exclusive

缓存之间的包含关系是重要的工程决策：

```
Inclusive（包含）:          Non-Inclusive（非包含）:        Exclusive（互斥）:
+------------------+       +------------------+       +------------------+
|       L3         |       |       L3         |       |       L3         |
| +--------------+ |       |                  |       |                  |
| |     L2       | |       |   L2   L2   L2   |       |   L2   L2   L2   |
| | +----------+ | |       |                  |       |                  |
| | |   L1     | | |       | L1 L1 L1 L1 L1 L1|       | L1 L1 L1 L1 L1 L1|
| | +----------+ | |       |                  |       |                  |
| +--------------+ |       +------------------+       +------------------+
+------------------+
  L3 包含 L1/L2 的       L3 可能包含也可能不        L3 与 L1/L2 内容
  全部数据              包含 L1/L2 的数据          完全不重叠
```

| 策略 | 特点 | 典型实现 |
|------|------|---------|
| **Inclusive** | L3 包含所有 L1/L2 的数据。一个地址如果在 L1/L2 中，必然也在 L3 中。优点是 L3 可以作为"侦听过滤器"（snoop filter）；缺点是浪费了 L3 容量（存了大量重复数据）。 | 较老的 Intel 架构 |
| **Non-Inclusive** | L3 不保证包含。一个地址在 L1/L2 中存在时可能在 L3 中也可能不在。这是现代处理器的折中选择。 | AMD Zen 系列、现代 Intel |
| **Exclusive** | 同一地址的数据只在一级缓存中存在。L1 驱逐的块放入 L2，L2 驱逐的块放入 L3。优点是利用了所有缓存的总容量；缺点是替换逻辑更复杂。 | AMD 部分架构 |

> **面试要点**：能说清楚 Inclusive 的一个关键优势 —— 当外部设备（或另一个 socket 的核）发起 snoop 时，如果地址不在 L3 中，就一定不在任何 L1/L2 中，省去了向所有核广播 snoop 的必要。

## 3. 缓存行概念

### 3.1 为什么需要缓存行

缓存的最小传输单位是 **缓存行（Cache Line）**，在 x86 和 ARM 架构上通常是 **64 字节**。

```
内存地址空间 (假设一个 int 是 4 字节):
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+
| 0 | 4 | 8 |12 |16 |20 |24 |28 |32 |36 |40 |44 |48 |52 |56 |60 |
+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+---+
|<-------------------- 一个缓存行 (64 字节) -------------------->|
  int int int int int int int int int int int int int int int int
  a   b   c   d   e   f   g   h   i   j   k   l   m   n   o   p
```

- 如果 CPU 访问 `int a`（地址 0），整个 64 字节的缓存行被载入 L1。
- 然后访问 `int b`（地址 4），已经在缓存行中，**L1 命中**。
- 这就是空间局部性的直接利用。

### 3.2 缓存行为什么是 64 字节而不是 8 字节？

如果缓存行只有 8 字节（一个字的宽度），L1 命中率会急剧下降 —— 每次访问一个新地址都需要向下一级缓存发请求。64 字节是一个经验上效果最好的平衡点：它大到足够捕获空间局部性，又没有大到浪费带宽（载入大量用不到的数据）。

### 3.3 伪共享问题预告

缓存行的存在引入了一个常见性能陷阱：**伪共享（False Sharing）**。当两个核心分别修改位于同一缓存行中的不同变量时，缓存一致性协议会反复地使对方的缓存行失效，导致性能剧烈下降。

```
+---------------------------------------------------------------+
| Cache Line (64 bytes)                                          |
| +-------+-------+-------+-------+-------+-------+-----+       |
| | var_a | var_b |       |       |       |       | ... |       |
| +-------+-------+-------+-------+-------+-------+-----+       |
+---------------------------------------------------------------+
    |         |
    v         v
 Core 0     Core 1
 写 var_a   写 var_b
  ==>  两者不断互相 invalidate，尽管访问的是不同的变量！
```

这是我们在后续课程中将会详细讨论的问题。

## 4. 缓存的内部组织

### 4.1 组相联（Set-Associative）结构

现代 CPU 缓存几乎全是 **N 路组相联（N-Way Set-Associative）**。缓存被分为若干组（Set），每组有 N 条路（Way）。一个内存地址被映射到唯一一组，但在该组内可以放在任意一路。

```
                         +-- Cache --+
                         |           |
    内存地址 ----------> | Set 0     |  Way 0  Way 1  Way 2  ...  Way 7
    映射到唯一的 Set     | Set 1     |  [====] [====] [====]       [====]
                         | Set 2     |  [====] [====] [====]       [====]
                         |  ...      |  [====] [====] [====]       [====]
                         | Set N-1   |  [====] [====] [====]       [====]
                         +-----------+
```

- **直接映射（Direct-Mapped）** = 1 路组相联：每个 Set 只有 1 个 Way。简单但冲突率高。
- **全相联（Fully Associative）** = 整个缓存只有 1 个 Set：数据可以放在任何位置。命中率高但硬件复杂、功耗大。
- **N 路组相联**：折中方案。实际 CPU 中 L1 通常是 8 路，L2 16 路，L3 可以达到 16-20 路。

### 4.2 物理地址的拆分

CPU 发出的物理地址在缓存查找时被解释为三个字段：

```
物理地址 (假设 48-bit):
+--------------------------------------------------+-----------+------+
|                   Tag                            | Set Index |Offset|
|                 (高位)                            |  (中间)   |(低位)|
+--------------------------------------------------+-----------+------+
  <------------------ t bits -------------------->  <-- s bits --><b bits>
```

| 字段 | 含义 | 示例（32KB 缓存，8 路，64B 行） |
|------|------|-------------------------------|
| **Offset** | 缓存行内的字节偏移。64 字节行需要 6 bits。 | bits [5:0] |
| **Set Index** | 选择哪个 Set。Set 数 = 缓存总大小 / (路数 x 行大小)。 | bits [s+5:6] |
| **Tag** | 剩下的高位，用于在 Set 内比对是否命中。 | bits [47:s+6] |

计算示例：32KB 缓存，8 路组相联，64 字节缓存行。

```
Set 数 = 32KB / (8 x 64B) = 32768 / 512 = 64 个 Set
Set Index bits = log2(64) = 6 bits
Offset bits = log2(64) = 6 bits
Tag bits = 48 - 6 - 6 = 36 bits
```

### 4.3 缓存查找过程 —— Tag SRAM vs Data SRAM

缓存由两块 SRAM 组成：

```
                    Tag SRAM                          Data SRAM
          +-------+-------+-------+         +-------+-------+-------+
  Set 0   | Tag 0 | Tag 1 | ... |Tag 7|     | Data 0| Data 1| ... |Data 7|
          | State | State |     |State|     |  (64B)| (64B) |     |(64B)|
          +-------+-------+-------+         +-------+-------+-------+
  Set 1   | Tag 0 | Tag 1 | ... |Tag 7|     | Data 0| Data 1| ... |Data 7|
          | State | State |     |State|     |  (64B)| (64B) |     |(64B)|
          +-------+-------+-------+         +-------+-------+-------+
          |  ...                             |  ...
          +-------+-------+-------+         +-------+-------+-------+
  Set 63  | Tag 0 | Tag 1 | ... |Tag 7|     | Data 0| Data 1| ... |Data 7|
          | State | State |     |State|     |  (64B)| (64B) |     |(64B)|
          +-------+-------+-------+         +-------+-------+-------+
```

> 注意：Tag SRAM 中除了 Tag 位，还存储 **状态位**（M/O/E/S/I —— 正是 MOESI 的状态）。这就是一致性协议与缓存硬件的结合点。

**查找流程（8 路组相联）**：

```
CPU 发出物理地址 A
        |
        v
   +----+----+----------------------------------+
   |  Tag   | Set Index (6b) | Offset (6b)      |
   +----+----+----------------------------------+
        |         |
        |         +--> 选中 Set (例如 Set 2)
        |                  |
        |         +--------v--------+
        |         | Tag SRAM: 并行读出    |
        |         | Set 2 的 8 个 Tag   |
        |         | + 8 个状态位         |
        |         +--------+--------+
        |                  |
        |    +-------------v-------------+
        |    | 比较器: 输入的 Tag 同时与 |
        |    | 8 个 Tag 并行比较        |
        |    +-------------+-------------+
        |                  |
        |    +-------------v-------------+
        |    | 哪一个 Way 的 Tag 匹配?    |
        |    | (且状态不是 Invalid)       |
        |    +-------------+-------------+
        |                  |
        |         +--------+--------+
        |         | 命中 (Hit)       |  --> 从对应的 Data Way 读出
        |         | 命中 (Hit)       |      数据，用 Offset 选择字节
        |         +-----------------+
        |         | 未命中 (Miss)     |  --> 向下一级缓存/内存发请求
        |         +-----------------+
        v
   (Tag 和 Set Index 用于查找，Offset 在命中后选择具体字节)
```

> **面试要点**：能解释为什么 Tag 比较是并行的 —— 为了在一个时钟周期内完成 L1 查找，硬件在 Set 内同时比较所有 N 个 Tag。这是组相联设计的关键代价：需要 N 个比较器。

## 5. 多核一致性问题

### 5.1 问题描述

假设两个核心各自有私有 L1 缓存。Core 0 写一个变量，Core 1 随后读同一个变量：

```
初始状态: x = 0 (在 DRAM 中)

时间线:
-------+-------------------------------+-------------------------------
       |        Core 0                |        Core 1
       |  (私有 L1: x=?)             |  (私有 L1: x=?)
-------+-------------------------------+-------------------------------
  t1   |  load x  --> L1 Miss         |
       |  DRAM 返回 x=0               |
       |  L1 缓存了 x=0               |
-------+-------------------------------+-------------------------------
  t2   |  store x=1 --> L1 中更新     |
       |  L1: x=1 (Dirty)             |  L1: ??? (不知道 x 被改了)
-------+-------------------------------+-------------------------------
  t3   |                               |  load x --> L1 Miss
       |                               |  DRAM 返回 x=0 !!! (如果没
       |                               |  有回写, DRAM 中仍是 0)
       |                               |  L1 缓存了 x=0
       |                               |  **读取到了过期的旧值!**
-------+-------------------------------+-------------------------------
```

### 5.2 可视化

```
   Core 0                           Core 1
   +------+                         +------+
   | L1 $ |  x = 1 (Dirty)          | L1 $ |  x = 0 (Stale!)
   +--+---+                         +--+---+
      |                                 |
      +--------+           +------------+
               |           |
          +----v-----------v----+
          |    L3 / DRAM         |
          |    x = 0 (旧值)      |
          +----------------------+
```

结果：**Core 1 读到的是过期数据**。这是一个数据正确性 bug，在任何多线程程序中都无法通过软件手段完全规避（除非你愿意放弃缓存带来的性能提升）。

### 5.3 一致性协议要解决的问题

缓存一致性协议（如 MOESI）的目标是：

1. **单写多读（Single Writer, Multiple Readers）**：对于同一个地址，任何时刻最多只能有一个核心在写。同时可以有多个核心在读，但不能有写者。
2. **写传播（Write Propagation）**：一个核心的写操作最终必须对其他核心可见。
3. **写串行化（Write Serialization）**：所有核心看到的对同一地址的写顺序必须一致。

> 这三个目标通常用 **SWMR + Write Propagation + Write Serialization** 来概括。MOESI 协议的每个状态转换都在服务这三个目标。

## 6. 写策略

### 6.1 Write-Through（写穿透/直写）

每次 CPU 写操作同时更新缓存和下一级存储（L2 或 DRAM）。

```
CPU 写 x=1:
    同时写入 L1      +      同时写入 L2/DRAM
    L1: x=1                  DRAM: x=1
       (Clean)                (最新)
```

**优点**：下一级存储始终是最新数据，一致性管理相对简单。

**缺点**：每次写都要等 DRAM 完成（100ns），**完全抵消了缓存的性能优势**。

### 6.2 Write-Back（写回）

CPU 写操作只更新缓存，并标记该缓存行为 **Dirty**。只有当该缓存行需要被替换（Evict）时，Dirty 数据才被写回下一级存储。

```
CPU 写 x=1:
    只写入 L1             DRAM: x=0 (旧值保留)
    L1: x=1 (Dirty)       

... 后续可能的多次写 ...

缓存行被替换时:
    L1 Dirty 数据 --> 写回 DRAM
    DRAM: x=1 (最终更新)
```

**优点**：大幅减少对下一级存储的写操作，性能好。多次对同一地址的写只会产生一次最终的回写。

**缺点**：一致性管理更复杂 —— 最新数据可能在某个核心的私有缓存中，DRAM 中的数据是旧的。需要协议（如 MOESI）来追踪。

### 6.3 为什么 Write-Back 是默认选择

| 维度 | Write-Through | Write-Back |
|------|-------------|-----------|
| 写延迟 | 高（必须等 DRAM） | 低（只写缓存，可以批量回写） |
| 带宽消耗 | 高（每次写都产生总线流量） | 低（只回写被替换的 Dirty 行） |
| 一致性复杂度 | 低 | 高（需要 Dirty 状态 + 协议） |
| 实际使用 | 极少数场景（某些嵌入式） | 几乎所有现代通用 CPU |

**结论**：现代 CPU 的 L1/L2/L3 缓存几乎全部使用 Write-Back + Write-Allocate（写不命中时先载入缓存行再修改）策略。这给缓存一致性协议带来了核心挑战 —— **最新数据可能在某个核的私有缓存中，而非在共享存储中**。

## 7. 课后测验

### Q1：为什么缓存行是 64 字节而不是 8 字节？

<details>
<summary>点击展开答案</summary>

**原因 1 — 空间局部性**：程序访问数据时，大概率也会访问相邻地址。64 字节的行一次载入可以覆盖多个连续变量或数组元素。如果是 8 字节，每次访问新地址都需要发起新的缓存请求，命中率会急剧下降。

**原因 2 — 摊销开销**：每次缓存访问需要比较 Tag、查找 Set。如果行太小，Tag SRAM 的开销占比过高（Tag 位数量 = Set 数 x 路数 x Tag 宽度，与行大小无关）。更大的行减少了 Tag 开销的相对百分比。

**原因 3 — DRAM 突发传输效率**：DRAM 本身也是按突发（Burst）传输的，一次 Row Activate 后可以连续读出多列数据。64 字节恰好匹配 DDR4/DDR5 的典型突发长度（BL8 x 64-bit = 64B）。

**经验平衡**：64 字节是工程实践中找到的甜点 —— 大到充分利用空间局部性和 DRAM 带宽，小到不过度浪费缓存容量在无用数据上。

</details>

---

### Q2：如果两个核心分别写同一缓存行中的不同变量，会发生什么？

<details>
<summary>点击展开答案</summary>

这会导致 **伪共享（False Sharing）**。

具体过程：

1. Core 0 写变量 A（地址 0x00），获得该缓存行的 **Modified (M)** 状态。
2. Core 1 写变量 B（地址 0x04，同一缓存行），一致性协议发 snoop 使 Core 0 的该行失效或降级。
3. Core 0 变为 **Invalid (I)** 或 **Shared (S)**。
4. Core 0 再次写 A 时，又需要重新获取所有权，再次使 Core 1 的该行失效。
5. 如此往复……

尽管两个核心操作的是 **完全不同的变量**，但由于它们位于同一缓存行（一致性协议的最小追踪单位是缓存行），硬件无法区分，导致双方的缓存行不断被互相 invalidate。这会造成显著的性能下降 —— 比真正共享同一个变量的情况可能更糟，因为产生了大量的总线流量和缓存 miss。

**解决方案**：通过填充（Padding）让高频写入的变量落在不同的缓存行上。例如在 C/C++ 中将变量对齐到 64 字节边界。这是后续课程中会深入的内容。

</details>

### Q3：在一个 32KB、8 路组相联、64 字节缓存行的缓存中，有多少个 Set？

<details>
<summary>点击展开答案</summary>

```
Set 数 = 缓存总大小 / (路数 x 缓存行大小)
       = 32 KB / (8 x 64 B)
       = 32768 B / 512 B
       = 64 个 Set
```

地址拆分：

```
Offset bits  = log2(64)   = 6 bits   (字节内偏移)
Index bits   = log2(64)   = 6 bits   (选择哪个 Set)
Tag bits     = 48 - 6 - 6 = 36 bits  (剩余高位用于匹配)

总计: 6 + 6 + 36 = 48 bits (典型的 48-bit 物理地址)
```

> **常见错误**：有人会忘记除以路数，直接用 32KB / 64B = 512。这 512 是"总共的缓存行数量"，而 Set 数需要在 8 条路之间分配。

</details>

### Q4：为什么 Write-Through 很少被实际使用？

<details>
<summary>点击展开答案</summary>

**核心原因：性能太差。**

1. **写延迟高**：每次 `store` 指令都需要等待数据写入下级存储（通常是 DRAM，延迟约 100ns）。这完全违背了缓存的设计初衷 —— 隐藏 DRAM 延迟。

2. **带宽消耗大**：每次写操作都产生向下的总线流量。对于写密集的程序，Write-Through 会迅速饱和内存带宽。

3. **能耗高**：频繁的 DRAM 写操作意味着更高的功耗（DRAM 每 bit 访问的能耗比 SRAM 高 1-2 个数量级）。

4. **Write Buffer 只是缓解而非解决**：虽然可以在缓存和下级存储之间插入 Write Buffer（CPU 写缓存后立即返回，Write Buffer 异步地把数据写入下级），但 Write Buffer 容量有限（通常只有几个条目）。一旦 Write Buffer 满了，CPU 仍然需要停顿等待。相比之下，Write-Back 从根本上减少了对下级存储的写次数。

**Write-Through 的唯一优势**是一致性管理简单（下级存储始终有最新数据），但在性能为王的通用 CPU 领域，这个优势远不足以弥补性能损失。

> **少数使用场景**：某些简单的嵌入式处理器、或者 L1 使用 Write-Through 但 L2 使用 Write-Back 的混合设计（e.g. 某些 ARM 实现），以及在调试/验证场景中简化推理。

</details>

### Q5：Inclusive 和 Exclusive L3 缓存有什么区别？

<details>
<summary>点击展开答案</summary>

**Inclusive L3**：
- L3 包含 L1 和 L2 中所有有效缓存行的副本。
- 如果某个地址在 L1/L2 中，则必定也在 L3 中。
- **最大优势**：L3 可以作为 snoop filter。当一个外部请求来查询某个地址时，如果该地址不在 L3 中，就可以确定它也不在任何 L1/L2 中，从而避免向所有核心广播 snoop 请求。这在多 socket 系统中尤其有价值。
- **最大劣势**：浪费 L3 容量。当 L1/L2 的总容量接近 L3 时（例如 16 核 x 32KB L1 x2 + 512KB L2 = 约 9MB），L3 中有大量容量被重复数据占用。如果 L1/L2 驱逐一行，L3 中对应的行也可能需要被 invalidate（为了维持包含性）。

**Exclusive L3**：
- L3 中的数据与 L1/L2 完全不重叠。一个地址一次只存在于某一级缓存中。
- L1 驱逐的块进入 L2（而不是被丢弃），L2 驱逐的块进入 L3（作为 victim cache 的角色）。
- **最大优势**：充分利用所有缓存的总容量。L1+L2+L3 的总有效容量 = 它们的和，没有重复。
- **最大劣势**：替换逻辑复杂（驱逐的数据需要"往下推"）；L3 不能作为 snoop filter（地址不在 L3 中不代表不在 L1/L2 中）。

**Non-Inclusive**：
- 现代处理器的折中。L3 不保证包含，L1/L2 中的数据可能在 L3 中也可能不在。
- 既有一定的灵活性，也保留了一定的 snoop filter 能力（可以在 L3 中维护一个目录来追踪）。

> **面试追问**：如果 Inclusive L3 中，L1 驱逐了一行，L3 需要做什么？答案 —— L3 需要将该行标记为"不在 L1/L2 中"（如果有精确的包含性追踪）。通常这不是把数据删掉，而是在 L3 的 Tag 位中维护一个位向量记录哪些核心的私有缓存中有这份数据。当所有核心都驱逐后，L3 可以让该行变成 exclusive 或被替换。

</details>

|--------|-----------|
| 存储墙 | CPU 比 DRAM 快 100 倍以上，缓存是唯一的桥梁 |
